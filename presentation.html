Challenge 2 — Discovery

What We Found

Three servers. Real problems.

  • !Python 2.7 — EOL since January 2020. Unpatched CVEs, no security fixes.
  • !Hardcoded credentials — DB password in a config.ini checked into version control.
  • !2am cron job — Batch reconciliation coupled to a shared NFS mount. If the mount drops, Finance doesn't know until morning.
  • ?Mystery dependency — http://10.0.1.45:8443/ledger/v1 — hardcoded IP, no docs, no service name. Nobody knows if it still exists.
  • !Postgres 11 — EOL November 2023. Running unpatched in production.

The real cost of standing still

$114K
per year, on-prem
Server hardware + power + rack + ops labor
Plus: unquantified breach risk from EOL software.
Plus: ~120h/yr manual patching across 3 servers.
Plus: 0 elastic capacity — can't scale down in slow periods.
Challenge 1 + 3 — The Memo & The Options

The Decision We Made

Option A — Rejected
Lift & Shift to EC2
Move servers to VMs. Keep the cron. Keep the NFS. Migrate the debt, don't fix it. Score: 14/20.
Option B — Chosen
ECS Fargate + RDS + S3
Containerize. Kill the cron. Refactor on the way in — Strangler Fig. Fix the debt, don't carry it. Score: 18/20.
Option C — Future State
EKS + Aurora
Right answer at scale. Wrong answer for a two-person team with a 3-hour clock. Score: 15/20.
The undocumented ledger API at 10.0.1.45 is a blocker in every option. We documented it, flagged it, and left an honest stub. That's the right call.

What We Actually Built

The innovations

  • 1
    Killed the cron job.
    S3 event notifications replace the 2am schedule. Files arrive → worker triggers. No polling, no missed runs, no shared filesystem.
  • 2
    Compatibility shim.
    Legacy consumers expect files at /mnt/reports/shared. The shim exposes the same interface over HTTP. Zero code changes downstream.
  • 3
    Runs locally today.
    Docker Compose with MinIO (S3), Postgres (RDS), Redis (ElastiCache). Same containers, same config, no AWS account needed to demo.
  • 4
    Tests define "done".
    35 tests across smoke, contract, and data integrity suites. Swap one env var — same tests run against AWS post-cutover.

Challenges 9 + 10 — Disaster Recovery & Rollback

When Things Go Wrong

Disaster Recovery

  • RTO15 minutes — Route 53 auto-failover (60s TTL) + ECS warm standby in us-west-2 + RDS replica promotion
  • RPO5 minutes — Continuous RDS binlog replication + S3 Cross-Region Replication
  • AutoRoute 53 DNS switch, CloudWatch alarm, SNS page — no human required for detection
  • ManualRDS replica promotion, SSM parameter update, ECS scale-up — 9-step numbered runbook

Migration Rollback

  • WebRoute 53 weight → 0 on ECS record. 2 minutes. One person.
  • BatchStop ECS service, run dedup SQL, re-enable on-prem cron, copy S3 files back to NFS. 20-30 minutes. Data engineer + DBA.
  • DBRe-point 5 Metabase instances to db-01.contoso.local. 75 minutes. DBA + PM.
  • WindowRollback window closes at 72h — criteria defined, not assumed. After that: forward-fix only.
